## §2 — The modular spine (catalog → endpoint → config)

This is the GamiPress-style backbone Adam built. The pattern is **clean where it's wired, bypassed in several places**. Below is the picture as it actually exists in code today.

### 2.1 The contract

```
┌──────────────┐    register     ┌────────────┐   reference    ┌────────────┐
│   ADMIN UI   │ ──────────────▶│  CATALOG   │◀────────────── │  ENDPOINT  │
│ /admin-panel │   via /api/    │ (DB table) │  via lib/* +   │  (API route│
└──────────────┘   admin/<x>    └────────────┘    SQL RPC     │  or worker)│
                                                              └────────────┘
                                                                     │
                                                                     ▼
                                                              ┌────────────┐
                                                              │ AUDIT LOG  │
                                                              │ + LEDGER   │
                                                              └────────────┘
```

A new feature **must** plug into one of the existing catalogs. New tables for new features bypass the spine.

### 2.3 The dispatch layer (the glue)

- **`lib/sysConfig.js`** — 60s in-memory cache + `setConfig`/`resetConfig` that append to `system_config_log`. THE library every server-side constant should be reading.
- **`lib/triggerEngine.js`** — `emitTrigger(triggerId, userId)` → `adminDb.rpc('process_trigger', ...)` → returns `[{special_id, fg_granted}]`. Sends grant emails async via `lib/sendGrantEmail.js`. Server-side only.
- **`process_trigger` PL/pgSQL function** — **NOT IN REPO** (lives in Supabase console, presumably created by the missing `migrations/027_quest_system_wiring.sql`). Reads `specials` matching the trigger, applies `special_rules`, awards FG/badges/XP, writes `special_claims`. The single most load-bearing piece of SQL in the system, and it's not in version control.
- **`lib/awardXp.js`** — XP add + rank promotion check + grant email side-effect. Uses `lib/rankEngine.js` thresholds.
- **`lib/rankEngine.js`** — 50-rank table + XP_REWARDS map. **Hardcoded** (CLAUDE.md says intentional — leaderboard integrity), but `RANK_XP[]` is also duplicated in `pages/api/admin/reconcile.js:40` with **divergent values** (Top-10 #3).

## §3 — Traced example: "Summon Forum Troll" quest, end-to-end

Tracing the canonical quest type Adam shipped this week (Patch 1/2/5). Every line below references real code.

### Step 1 — Admin creates the quest

The admin opens the Quests tab in `/admin-panel`. UI POSTs to `pages/api/admin/quests.js`. The route writes a row to the `quests` table:

```jsonc
{
  id: "<uuid>",
  name: "Summon Forum Troll",
  type: "one_time",                // or 'daily' | 'weekly'
  trigger_id: "forum_troll_spawned",
  xp_reward: 35,
  fg_reward: 50,
  requirements: { /* threshold etc. */ },
  rewards: { /* granted via emitTrigger */ },
  completion_text: "A troll has spawned in the forums!",
  active: true,
  config: {                        // quest-owned params
    clicks_to_kill: 3,
    despawn_minutes: 30,
    spawn_location: "any",         // 'ladder' | 'eternal' | 'd4' | 'd2r' | 'd3'
    spawn_limit: 20
  }
}
```

### Step 2 — Trigger registered

The `triggers` table already has a row `forum_troll_spawned` (admin pre-seeded; managed via `/api/admin/trigger-config.js`). Its config:

```jsonc
{
  expires_after: 7,                // days; null/missing = no expiry
  cycle_start: "2026-04-26T...",   // updated each cycle for recurring
  recurring: true,
  on_expiry: "auto_fire",          // or 'deactivate'
  allowed_subscriptions: ["premium", "legendary"]   // or [] = all
}
```

### Step 3 — Specials attached

Optionally, `specials` table rows attach FG/badge grants to this trigger:

```jsonc
// specials row
{
  id: "<uuid>",
  trigger_id: "forum_troll_spawned",
  fg_grant: 100,
  badge_id: null,                 // could grant a badge
  enabled: true
}
```

`special_rules` rows can add extra conditions (e.g. only on the user's first spawn).

### Step 4 — User clicks the gem

`AppShell.js` listens for the troll click. It POSTs to `pages/api/quest-trigger.js`:

```js
fetch('/api/quest-trigger', {
  method: 'POST',
  headers: { Authorization: 'Bearer ' + token },
  body: JSON.stringify({ trigger_id: 'forum_troll_spawned' })
})
```

### Step 5 — `quest-trigger.js` orchestrates

`pages/api/quest-trigger.js:22-188`:
1. Verifies JWT → `user.id`.
2. Reads `triggers` row, checks `enabled`, `expires_after`, `allowed_subscriptions` against the user's `subscription_tier`.
3. Loads all `quests` rows where `trigger_id = 'forum_troll_spawned'` and `active = true`.
4. For each quest: upserts `quest_progress`, marks `completed=true` when threshold hit (currently hardcoded to `1` — see §4 finding M-3).
5. On completion, calls `_grantQuestRewards()` which adds `fg_reward` + `xp_reward` to the `users` row inline (**bypasses `fg_ledger` and `awardXp.js` rank-up flow** — see §4 finding M-4).
6. If `trigger_id === 'forum_troll_spawned'`, calls `_spawnForumTroll()` which inserts into `forum_trolls` table with HP/despawn/thread_id from `quest.config`.
7. Calls `emitTrigger('forum_troll_spawned', user.id)` from `lib/triggerEngine.js` → `process_trigger` SQL RPC → fires all `specials` matching the trigger and writes `special_claims` rows.
8. Resets `cycle_start` if `triggers.config.recurring`.

### Step 6 — Realtime + UI

`forum_trolls` is in the `supabase_realtime` publication (added today by `migrations/038_forum_trolls_realtime.sql`). Every connected client gets a `postgres_changes` INSERT event. `AppShell.js` and `HomeView.js` rerender:
- Banner appears
- Gem stays pressed (`gem-pressed` CSS class with `!important`, Patch 5)
- The chosen thread shows the troll on its card

### Step 7 — Eventual cleanup

`pages/api/admin/trigger-expiry-check.js` is invoked (admin-only POST, **no real cron yet — needs external scheduler**). It scans `triggers` with `expires_after`, computes `cycle_start + days`, and on expiry either re-fires (`on_expiry='auto_fire'`) or disables (`on_expiry='deactivate'`). Recurring triggers reset `cycle_start`.

### What this trace tells us about the contract

- **Quest config is split between `triggers.config` and `quests.config`** by design. Trigger config = "when does this fire and to whom." Quest config = "what happens when it fires." That distinction is good and consistent across the few quest types implemented.
- The `process_trigger` SQL function is the single most load-bearing piece of SQL. **It is not in the repo.**
- Reward payout has **two parallel paths** — `_grantQuestRewards()` (inline `UPDATE users SET fg_balance = fg_balance + ?`) AND `emitTrigger() → process_trigger → fires specials`. They are not transactional with each other. A user who hits a quest with both `quest.fg_reward = 50` AND a `special` granting `100` gets two separate UPDATEs. If only one succeeds, the user is partially rewarded silently.

## §4 — Findings by severity

Severity scale: **CRITICAL** (silent breakage at launch / money loss vector) · **HIGH** (real risk under normal load) · **MEDIUM** (smell, correctness drift, fragile) · **LOW** (nit, future-proofing).

### CRITICAL

**C-1 · Unauthenticated admin seed endpoint.** `pages/api/admin/seed-bot-threads.js:121` deletes all GoblinBot threads and inserts 5 fixed bot threads. **Zero auth check.** `curl -X POST https://trade.d4jsp.org/api/admin/seed-bot-threads` works from anyone. Adds bot threads to your live feed and silently deletes existing ones. → Add `requireAdmin(req)` exactly like `pages/api/admin/badges.js` does.

**C-2 · Token-only seed endpoint.** `pages/api/admin/seed-rewards-once.js:7` only checks `req.query.token === 'd4jsp-reward-seed-2026'`. The string is in the source code (which is on the VPS, in git, in this audit). Anyone running `curl '...?token=d4jsp-reward-seed-2026'` inserts arbitrary `skills` rows. → Either add `requireAdmin` or delete the file (the comment at line 1 already says "delete after seeding").

**C-3 · Rank XP threshold divergence.** `lib/rankEngine.js:1-52` defines 50 ranks with thresholds. `pages/api/admin/reconcile.js:40` defines an inline `RANK_XP[]` array of **40 values that diverge starting at rank 30** (286,700 vs 286,600 in `rankEngine`). `getRankId()` in `reconcile.js` is the function that decides which rank's `rank_grants` skills a user receives — so for any user past rank 29 the wrong skills are granted. → Replace the inline array with `import { RANKS } from '../../../lib/rankEngine'; const RANK_XP = RANKS.map(r => r.xp);`. Then run `reconcile_all`.

### HIGH

**H-1 · `.env.production` committed with secrets.** `git ls-files | grep env` shows it tracked. `.env.production:14-15` contains `NEXT_PUBLIC_VAPID_PUBLIC_KEY` and `VAPID_PRIVATE_KEY` in plaintext. Per `CREDENTIAL_ROTATION_INVENTORY.md` Section C, the file has been modified across 5 commits. Web-push private key compromise = signed push notifications to all subscribers. → Rotate VAPID, `git rm --cached .env.production`, add `.env.production` to `.gitignore`.

**H-2 · `.env.production.kvm4` on disk with full live secrets.** Untracked but easy to commit by accident: contains `SUPABASE_SERVICE_ROLE_KEY`, `STRIPE_SECRET_KEY`, `STRIPE_WEBHOOK_SECRET`, `BATTLENET_CLIENT_SECRET`, `RESEND_API_KEY`, `VAPID_PRIVATE_KEY`. **The `.gitignore` does NOT contain a wildcard `.env*` pattern** — only `.env.local` and `.env`. → Add `.env*` to `.gitignore`.

**H-3 · Non-atomic FG transfer.** `pages/api/transfer-fg.js:36-46` does `UPDATE users SET fg_balance = X WHERE id = sender` then `UPDATE users SET fg_balance = Y WHERE id = recipient`. No transaction. Crash, network error, OOM, deploy-mid-request between them → FG missing. The `transfer_fg` PL/pgSQL function in `docs/numbered-fg-vault.md` is the proposed fix and is not yet implemented. → Implement that RPC; replace both UPDATEs with a single `adminDb.rpc('transfer_fg', {...})`.

**H-4 · Two parallel admin endpoints, only one audited.** `pages/api/admin/action.js` (legacy) handles `banUser`, `unbanUser`, `setDisplayName`, `setGold`, `setMembership`, `setBadge` with NO `admin_action_log` writes, NO `fg_ledger` writes. `pages/api/admin/user-detail.js` (new) handles the same actions correctly. Both are wired from AdminView depending on tab. After launch, half of admin actions vanish from forensics. → Migrate AdminView to call `user-detail.js` only, delete `action.js`.

**H-5 · `banByIp=true` collateral.** Both `pages/api/admin/action.js:28-35` and `pages/api/admin/user-detail.js:407-414` ban every user sharing the target's `ip_address`. Shared WiFi, CGNAT, VPN, ISP NAT all cause collateral bans. Same finding from `FINDINGS.md` H1 — still applies. → Either remove the option, or require an admin re-confirmation (`{ banByIp: true, confirm_collateral_ips: 'YES' }`).

## §5 — Hidden coupling & "easy to break" map

These aren't findings, they're **load-bearing seams** the next round of fine-tuning has to respect.

### 5.1 Cross-domain auth chain (touch with care)

`.d4jsp.org` cookie (chunked Supabase session) + `d4jsp_auth` JWT cookie tie together:
- main trade app (KVM 4)
- admin panel (KVM 2, basePath `/admin-panel`)
- 7 WordPress satellite sites (Cloud)
- WP `d4jsp-supabase-auth` plugin (lives only on Cloud filesystem)

Files involved:
- `lib/supabase.js` (chunking, cookie name scheme, `CHUNK_SIZE = 2800`)
- `lib/auth-context.js:10-18` (`syncAuthCookie`)
- `lib/auth-context.js:300-315` (`logOut` → `https://d4jsp.org/d4jsp-logout`)
- WP plugin (not in repo)

**Touch one without the others and people get logged out across 8 sites.** Don't.

### 5.2 RLS depends on code shape

`migrations/019_users_rls_column_guard.sql` lists exactly which columns are protected on `users`. Adding a new sensitive column (e.g. `is_moderator`, `vip_status`) without updating 019 leaves it world-writable from any signed-in JWT. Flag for any future `ALTER TABLE users ADD COLUMN`.

### 5.3 The two reward payout paths

Quest completion fires:
- inline `_grantQuestRewards()` (FG/XP added directly to `users`, no ledger)
- `emitTrigger() → process_trigger SQL → specials engine` (FG/badge/XP via `special_claims`)

These run sequentially without a transaction. If one fails, the other has already happened. Until M-2 and M-4 are addressed, treat reward payout as best-effort.

### 5.4 OCR sell pipeline

Single point of failure on KVM 2:9000. Stack: AppShell.js `_doOcrFetch` (20s timeout) → `/api/paddle-ocr` (55s server abort) → KVM 2 RapidOCR (30s per OCR call). If KVM 2 is down, sell flow breaks; user sees error after wall-clock 90s. The `_findItemInDB` chain queries `wowhead_tooltips` AND `d4_equipment` AND `d4_items` — all `d4_*` tables have permissive RLS per `docs/security-audit-pass-2.md` W-10/11/12 ("DON'T TOUCH OCR rule"). **Tightening any `d4_*` RLS will silently break the sell pipeline.**

### 5.5 Realtime publication

`migrations/038_forum_trolls_realtime.sql` was applied today. **Other tables likely missing from the publication**: e.g. `quests` (admin updates a quest, no propagation), `triggers` (config changes don't propagate). For any new realtime subscription the table must be added with `ALTER PUBLICATION supabase_realtime ADD TABLE <name>`.

### 5.6 Admin app drift

Per `start.md` and `ACCESS.md`: D4JSP-Admin GitHub repo is STALE; admin code is edited in-place on KVM 2 at `/opt/d4jsp-admin`. **There is no version control for admin changes.** A reboot that wipes the box loses work that hasn't been mirrored back. Either re-establish the repo as source of truth or take periodic `tar` snapshots.

### 5.7 Two configuration concepts called "config"

- `system_config` — operational tunables (escrow.lock_hours, …) → `lib/sysConfig.js`
- `site_config` — older legacy key/val (bot_global_enabled, last_bot_activity, bot_auto_activity) → ad-hoc reads
- `bot_configs` — per-bot personality
- `triggers.config` jsonb — per-trigger config
- `quests.config` jsonb — per-quest config
- `subscription_tiers.skills_config` jsonb
- `rank_grants.config` jsonb
- `config_fields` — schema of `system_config` keys

This is fine; just be deliberate which one a new config belongs in. **`system_config` is the catalog for operational tunables**. New tunables belong there, not in code, not in `site_config`.

### 5.8 The 7,500-line AdminView

Anything you change in admin runs the risk of breaking another tab via shared state, shared fetchers, shared modals. This is the most fragile single file in the repo. Until the rest of `AdminView.js` is split (`L-6`), changes inside it should be small and well-tested by hand.

They share **one Supabase project** (`isjkdbmfxpxuuloqosib`) and one cross-domain auth cookie. The integration is mostly through that DB and a few proxy rewrites — there is no internal RPC bus.

### 7.2 What crosses each boundary

| Edge | Mechanism | Data |
|---|---|---|
| Trade app ↔ Supabase | `@supabase/supabase-js` 2.100, anon JWT (client), service-role key (server) | Everything: catalogs, users, threads, ledger, realtime |
| Trade app ↔ KVM 2 OCR | HTTP fetch (`PADDLE_OCR_URL`) | Base64 image → OCR text array |
| Trade app ↔ KVM 2 Tooltip | HTTP fetch (`VPS_TOOLTIP_URL`) | Wowhead URL → screenshot PNG to Supabase Storage |
| Trade app ↔ Admin app | Nginx → KVM 2:3001 (rewrites in `next.config.js:101-107`) | HTML pages + assets at `/admin-panel/*` |
| Admin app ↔ Trade app | Reverse rewrites (`D4JSP-Admin/next.config.js:14-25`): `/api/*` → `${TRADE_URL}/api/*` | All admin API calls go BACK to the trade app's `/api/admin/*` routes — admin app has zero of its own API logic |
| Admin app ↔ Supabase | Direct anon-client use of `lib/supabase.js` (login + JWT only) | Auth session |
| Build Planner ↔ Supabase | Direct anon-client use of `lib/supabase.js` | Reads `wowhead_tooltips`, writes `user_builds` via `/api/save-build` on trade app |
| Build Planner ↔ Trade app | Embedded into trade app at `/builder` (static export, basePath `/builder`); component lazy-loaded inside trade-app build | Permission gate via `admin_permissions.d4_build_slots` |
| Map app ↔ anything | None — fully static. localStorage for build rotations. | None |
| Map app ↔ Trade app | Embedded as iframe in Profile tab (per recent commit `520ab1b feat(profile): embed Map iframe in Character tab`) | iframe URL only |
| Trade app cookie ↔ Admin app cookie ↔ WP plugin | `.d4jsp.org` cookie + `d4jsp_auth` JWT cookie | Single sign-on across the federation |

  - **A-5 (LOW):** `next.config.js:38` defaults `NEXT_PUBLIC_ADMIN_EMAIL` to `adam87lewis@gmail.com`. Same as trade app M-12.

### 7.4 The 4-app rules of engagement (don't break these)

1. **All four apps point at one Supabase project.** Do NOT spin up a second project for testing without copying the entire schema (and the missing `process_trigger` function). Use Supabase branches if you need isolation.
2. **Auth is the cross-domain cookie chain.** Touching `lib/supabase.js` cookie storage in any of the four apps requires updating ALL four — the trade app, admin app, and Build Planner all import their own `lib/supabase.js`. Map has no auth so it's exempt.
3. **Admin code edits live in-place on KVM 2** until the GitHub repo is re-established. Treat `/opt/d4jsp-admin` as source of truth there.
4. **Build Planner is the model citizen.** New features: copy that migration pattern. Catalog row → tier-mapped JSON → endpoint reads via `admin_permissions`.
5. **Map is iframed in.** Don't add Supabase calls to it; if you need persistence, hand the data off to the trade app via a fetch.

### 7.5 Connected-systems findings (additions to §4)

- **CS-1 (CRITICAL):** No single source of truth for the database schema. Migrations are split across `D4JSP/migrations/` (13 files, gaps) and `D4JSP-Build-Planner/migrations/` (2 files). Plus undocumented changes applied directly in the Supabase console. → Establish a single canonical `db/migrations/` directory; symlink or import from each app.
- **CS-2 (HIGH):** Admin app drift (A-1 above) — admin code edited on KVM 2 outside git. Highest single-point-of-failure in the system.
- **CS-3 (HIGH):** Code duplication between trade app and admin app (`lib/auth-context.js`, `lib/rankEngine.js` minimum, possibly `AdminView.js`). Already drifted: trade app's auth-context was hardened for privilege escalation; admin's may not be. → Either turn admin into a 100% proxy app (no shared lib) or extract into a shared `packages/shared` with proper imports.
- **CS-4 (MEDIUM):** Build Planner is on Next.js 14 while trade app is on 15. Working today; will silently break on the day they need to share a server-side dependency. → Bump Build Planner to 15.
- **CS-5 (MEDIUM):** Map app's localStorage builds vs trade app's `user_builds` table — two divergent persistence stories for the same user concept. → Promote Map's local builds into `user_builds` so both apps see the same data.
- **CS-6 (LOW):** `D4JSP-Build-Planner/.claude/worktrees/` has 9 stale worktrees; `D4JSP/.claude/worktrees/` has the active one this audit ran in. Not a bug; flag for tidiness.
